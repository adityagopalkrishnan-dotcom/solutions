Usecases:
At QuestionPro, middleware requests typically fall into four categories: ETL integrations for syncing data with external systems, SMS integrations for sending surveys or running conversational surveys, WhatsApp integrations for interactive surveys using CMILES, and change requests, often involving updates to logic or providers—most commonly, requests for log data dumps for audit or debugging.


1. ETL Integrations (Extract, Transform, Load)
What it does:
Facilitates the movement of data to and from QuestionPro, enabling customers to keep their internal systems in sync with survey data or to trigger surveys based on business events.


Common Scenarios:


* Exporting survey responses to CRMs (e.g., Salesforce, HubSpot) or data warehouses 
* Importing contact lists, transactions or metadata from internal databases into QuestionPro for targeted surveys.
* Scheduled or event-driven sync jobs / Automated daily or weekly ETL pipelines.
Key Features:
* Custom data mapping and transformation
* Scheduled or real-time pipelines
* Logging, retries, and audit trails
2. SMS Integrations
What it does:
Enables survey distribution and conversational surveys via third-party SMS providers.
Common Scenarios:
* Sending transactional or post-interaction surveys via SMS.
* Full conversational SMS surveys (one question per message).
* Automating reminders and follow-ups via SMS.
________________
3.  WhatsApp Integrations
What it does:
Leverages WhatsApp Business APIs via CMILES to enable both survey invites and conversational surveys via the world's most used messaging app.
Common Scenarios:
   * Trigger-based WhatsApp surveys (e.g., case closed, delivery completed).
   * Running NPS or CSAT surveys inside WhatsApp.
   * Interactive survey flows using buttons, quick replies, and templates.


4. Change Requests and Enhancements
What it covers:
Ongoing modifications and refinements to existing integrations, often requested post-deployment to better align with evolving business needs or to improve reliability.
Common Scenarios:
   * Updating field mappings or workflows in ETL jobs
   * Switching SMS/WhatsApp providers
   * Adding custom logic (e.g., conditional triggers, language preferences)
   * Generate and share data dumps of integration logs (frequent request for audit, compliance, or debugging)

Internal Process for Raising a TechOps Request for Middleware


   * The Account Manager (AM) or Customer Success Manager (CSM) is responsible for creating a new TechOps ticket for any Middleware-related requests with a detailed process flow.
   * Once the ticket is created, the Middleware team is responsible for sharing the timeline required for the implementation after gathering all the necessary requirements and implementing it.


Important note


The implementation is handled by QuestionPro. Based on the level of effort required, the process typically takes 2 to 3 months to complete for a new implementation/integration from scratch and 7 to 30 days for enhancements on existing implementation.




________________
Implementation Process Flow
   1. Requirement Gathering
   * Initial discussion to understand business objectives and desired outcomes.
   * Define scope and platform integration needs (if any).
   * Middleware team would need to come back with the process flow design based on requirements.
   2. Configuration & Setup
   * The QuestionPro team sets up the necessary configurations in the platform. (Survey design, webhooks etc.)
   * For any third party systems or integrations required, QuestionPro team needs to be provided with the required technical details (Credentials, API documentation etc.)
   3. Development
   * After technical review, an ETA will be shared by Pavan or Priyesh
   * Any requirement changes may effect the ETA and will be shared accordingly.
   4. Review & Feedback
   * A preliminary version is shared with the client for review.
   * The client provides feedback or requests changes if needed.
   5. Revisions & Finalization
   * The QuestionPro team incorporates revisions.
   * The final version is reviewed and approved by the client.
   6. Testing & Quality Assurance
   * Internal testing is performed to ensure functionality and data accuracy.
   * A test run may be provided for client verification.
   7. Go-Live
   * Once approved, the implementation will be deployed.
   * Monitoring and support are provided post-launch as needed.

What can be done
On a high level middleware addresses the following:
      1. Data exchange between external systems (client’s FTP servers, SalesForce etc) and QuestionPro.
      2. Integration with external systems to pull and push data.
      3. Data transformation
Sample Scenarios
Below are some examples of different requirements that are handled by the middleware team. Depending on whether it’s a net new requirement or change request for existing middleware, the timeline and costs vary.
Net new requirements
      1. The client wants contacts to be fetched from the SFTP server, which needs to be processed in middleware and then surveys to be triggered via WhatsApp and SMS.
      2. The client wants to integrate with SMS provider for two-way SMS communication to collect the feedback and push the response into the QuestionPro system
      3. Send WhatsApp messages to users on upload of data and a reminder after 24 hours to participants who did not respond.
      4. The client wants a setup where surveys are automatically triggered from Salesforce via Outbound Messages based on flow criteria (e.g., case closed or opportunity won).
      5. The client needs the survey responses to be stored in an SQL database, which will be used by their BI tool for data visualization.
Change requests/Feature Enhancements
      1. The client wants the “Last name” to be fetched from Salesforce and passed to the survey as a custom variable.
      2. Implementation of duplicate SMS prevention functionality to avoid sending multiple SMS messages to the same mobile number during survey distribution campaigns.
      3. In addition to the existing surveys being sent via WhatsApp and SMS, the client needs to start sending the new surveys as well.

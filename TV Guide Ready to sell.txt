1. What problem does this solution solve? (1-2 sentences)

Customers with large or frequently changing catalogues cannot rely on long static answer lists without hurting respondent experience, data quality and completion rates.

The Dynamic TV Guide Engine solves this by turning a standard QuestionPro survey page into a searchable selection experience, where respondents find the right item, add it to a personal list and then answer follow-up questions about exactly what they selected.

2. What is the solution and how does it work?

What it does

Dynamic TV Guide Engine is a services-led implementation pattern that converts QuestionPro survey questions into a searchable guide-style interface. Respondents can search by title, episode title, synopsis or description, add items to a My Programmes or My Podcasts basket, remove items, and continue to downstream questions that reuse their selections.

How it fits into the existing product

The solution sits on top of QuestionPro Survey, QuestionPro Middleware Datasets, custom variables, piping and advanced JavaScript. Middleware stores the uploaded catalogue as a versioned dataset and exposes it through a Dataset API endpoint. In production, a secure proxy can sit between the survey and Middleware so the production API key is handled server-side.

Core components

  --------------------------------------------------------------------------------------------------------------------------------------------------------
  Component                      Role in the solution
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------
  QuestionPro Survey UI          Hosts the respondent-facing search fields, result list, selected basket, validation and downstream questions.

  Middleware Dataset             Stores the searchable programme/catalogue data from CSV or Excel uploads and exposes the dataset endpoint.

  Secure proxy pattern           Recommended for production so the browser does not expose the Middleware API key.

  Search basket                  Lets respondents add, remove and de-duplicate selected programmes or catalogue items.

  Custom variables               Stores selected values into controlled ranges such as custom101-custom125, custom126-custom150 and custom151-custom175.

  CID / internal ID tagging      Preserves ContentID or another internal identifier in the stored value while stripping it from visible text.

  Pre-fill and cleanup scripts   Restores selected values when respondents navigate back and removes technical tags from the respondent-facing UI.
  --------------------------------------------------------------------------------------------------------------------------------------------------------

When/how a customer would use it

Customers use this when they need respondents to choose from a large dataset, but also need those choices to drive survey logic, follow-up questions, exports or analytics matching. The TRP Ireland proof point applies this to programmes, streaming content and podcasts, but the same pattern can apply to products, stores, sessions, accounts, branches, facilities, events or any other searchable reference dataset.

3. What are the key benefits? (3-5 bullets)

  · Improves respondent experience by replacing long static lists with a fast, searchable, guide-like interface.

  · Supports large and changing catalogues without rebuilding the survey every time the dataset changes.

  · Keeps downstream QuestionPro logic intact by writing selections into custom variables that can be piped, looped and exported.

  · Preserves backend identifiers such as ContentID for analytics matching while keeping the respondent-facing text clean.

  · Creates a reusable services-led pattern for media research, product feedback, branch selection, event/session feedback and other large-reference-list use cases.

4. Who is it for? (Target users and use cases)

Roles that care

  · Media research and audience measurement teams

  · Market research and insights teams

  · CX / VoC teams that need respondent-level context

  · Product, catalogue and operations teams

  · Data and analytics teams that need stable IDs behind selected labels

Workflows it improves

  · Weekly TV, streaming or podcast tracking surveys

  · Product catalogue feedback studies

  · Store, branch, location or venue selection in CX surveys

  · Training-session or event feedback surveys where each session has a unique ID

  · B2B surveys where respondents must select accounts, products, facilities or touchpoints before follow-up questions

Most useful when

  · The answer list is too large for a normal multi-select question.

  · The dataset changes regularly and should be updated outside the survey build.

  · Respondents may remember only part of the item name or description.

  · The selected item must carry a hidden backend identifier for reporting or matching.

  · Selections need to be reused later in the survey through piping, loops or custom variables.

Great fit vs. poor fit

5. How do you demo it?

Recommended internal demo flow:

  1. Start with the pain: show how a normal static list becomes unusable when the respondent has to choose from a large catalogue.

  2. Open Middleware > My Datasets and show the uploaded dataset with fields such as ContentID, Title, Series, Episode, EpisodeTitle, Synopsis, SortOrder and ViewingType.

  3. Explain that the survey calls the Middleware Dataset API endpoint, or a secure proxy in production, from the searchPrograms() layer.

  4. Launch the survey and type a partial title or synopsis into the search field to show real-time search after the 250ms debounce.

  5. Click a result and add it to My Programmes / My Podcasts. Show that selected items disappear from the result list to prevent duplicates.

  6. Remove an item and show that it becomes searchable again.

  7. Submit the page and move to a downstream question where the selected programme appears as clean piped text.

  8. Show the backend/export logic: the stored value can retain ContentID/CID for matching while the respondent sees only the clean label.

  9. Close by positioning this as a reusable pattern for any large, changing reference dataset, not just media content.

Suggested demo talk track

"Instead of forcing respondents to scroll through a long list, we turn the survey into a searchable guide. The respondent searches for what they remember, adds the right item to their personal basket, and the survey then uses those selections for follow-up questions. On the backend, we still preserve the item ID, so analytics can match every response back to the correct programme, product or record."

7. Competitive angle

  · Many survey tools can display static answer lists, but they do not easily create a true searchable basket experience inside the survey flow.

  · Custom web apps can solve the UI problem, but they usually break the native survey workflow, reporting and respondent-level data structure.

  · QuestionPro can keep the entire experience inside the survey while still using Middleware datasets, custom variables, piping and exports.

  · The differentiator is not only search. It is search plus selected-basket logic plus hidden ID preservation plus downstream survey reuse.

  · This gives teams a practical way to modernize complex surveys without forcing respondents into a separate portal or asking customers to rebuild their research workflow.

9. FAQ (3-5 questions max)

Is this a standard QuestionPro feature?

  No. Position it as a services-led implementation pattern built using QuestionPro Survey, Middleware Datasets, advanced JavaScript, custom variables and optional secure proxy development.

Can the customer update the programme or catalogue data themselves?

  Yes, if the dataset structure remains stable. The customer or Services team can update the CSV/Excel source or Middleware dataset version without rebuilding the full survey UI. Field names should remain consistent.

Will the API key be visible in the survey?

  In a controlled internal demo, the survey can call the Middleware endpoint directly. For production or customer-facing deployments, use a secure proxy so the API key is stored server-side and not exposed in browser-side JavaScript.

Can this work outside media research?

  Yes. The pattern works for any large searchable reference dataset: products, branches, properties, events, training sessions, accounts, facilities, assets or service categories.

What are the main implementation risks?

  The main risks are mismatched field names, reused custom-variable ranges, API key exposure, missing ContentID, insufficient QA on downstream piping, and dataset changes that break the expected schema.

Appendix: Quick qualification checklist

Use these questions during discovery to decide whether this solution is a fit:

  · How many options/items does the respondent need to search or select from?

  · How often does the dataset change?

  · Which fields should be searchable: title, description, ID, category, location, date or another field?

  · Does each selected item need to carry a backend ID for reporting or matching?

  · Which downstream questions need to reuse the selected values?

  · What custom variable range can be reserved for this flow?

  · Will the deployment require a secure proxy to keep API keys server-side?

  · Who owns dataset updates after go-live?
